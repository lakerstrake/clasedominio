\section{Introducci\'{o}n}

Este documento es la soluci\'{o}n al taller que nos puso la instructora Diana Delgado sobre Spring Boot CRUD. Lo que ten\'{i}amos que hacer era tomar el proyecto que ya ven\'{i}amos trabajando en clase, que es una aplicaci\'{o}n web para gestionar individuos hecha con Spring Boot, Thymeleaf y PostgreSQL, y hacerle unas mejoras espec\'{i}ficas.

Las mejoras que me tocaba hacer eran: agregar la columna de \textbf{tel\'{e}fono} a la tabla, \textbf{separar} las acciones de Ver, Editar y Eliminar, poner un \textbf{bot\'{o}n de ``Nuevo Individuo''}, y comprobar que se pueda \textbf{crear, editar y eliminar} desde la interfaz verificando que se refleje en la base de datos.

Tambi\'{e}n ten\'{i}a que explicar con mis propias palabras qu\'{e} hace cada capa del proyecto.

% ====================================================================
\section{Objetivo}

Mejorar la aplicaci\'{o}n CRUD que hicimos en clase con Spring Boot, Thymeleaf y PostgreSQL, agreg\'{a}ndole las funcionalidades que pide el taller y demostrando con capturas que todo funciona.

\section{Qu\'{e} Hace Cada Capa del Proyecto}

El proyecto usa la arquitectura MVC (Modelo--Vista--Controlador). Cada capa tiene un trabajo espec\'{i}fico y as\'{i} el c\'{o}digo queda organizado. A continuaci\'{o}n explico cada una.

\subsection{Entity -- Individuo.java}

Esta es la clase que representa a una persona en el sistema. Tiene los datos como nombre, apellido, edad, correo y tel\'{e}fono. Cada atributo se convierte en una columna de la tabla en la base de datos gracias a la anotaci\'{o}n \texttt{@Entity}. Yo lo veo como el ``molde'' de los datos: le dice a la app qu\'{e} informaci\'{o}n va a guardar de cada individuo. Tambi\'{e}n tiene \texttt{@Id} para que cada registro tenga su propio n\'{u}mero \'{u}nico que se genera solo.

\imagen{1}{C\'{o}digo de Individuo.java -- la entidad que representa la tabla en la base de datos.}

\subsection{DAO -- IndividuoDao.java}

El DAO es lo que conecta la aplicaci\'{o}n con la base de datos. Es una interfaz que extiende de \texttt{CrudRepository}, y lo bueno es que Spring ya trae listos los m\'{e}todos para guardar, buscar, listar y borrar sin tener que escribir SQL. Tambi\'{e}n le agregu\'{e} un m\'{e}todo para buscar por nombre.

\imagen{2}{C\'{o}digo de IndividuoDao.java -- interfaz que conecta con PostgreSQL.}

\subsection{Service -- IIndividuoService.java e IndividuoService.java}

El servicio est\'{a} en medio del controlador y el DAO. Hice una interfaz (\texttt{IIndividuoService}) donde puse los m\'{e}todos que necesitaba: listar, buscar, guardar y eliminar. Despu\'{e}s la clase \texttt{IndividuoService} implementa esos m\'{e}todos llamando al DAO. La idea de usar interfaz es por si toca cambiar algo despu\'{e}s, solo se cambia la implementaci\'{o}n y ya.

\imagen{3}{C\'{o}digo de la interfaz IIndividuoService.java y su implementaci\'{o}n IndividuoService.java.}

\subsection{Controller -- HomeController.java}

El controlador es como el ``cerebro'' de la app: recibe lo que el usuario hace en el navegador y decide qu\'{e} mostrar. Si entro a \texttt{localhost:8080/} me muestra la lista, si voy a \texttt{/nuevo} me abre el formulario, y as\'{i} con cada ruta. Tambi\'{e}n tiene las rutas de \texttt{/guardar}, \texttt{/editar}, \texttt{/eliminar} y \texttt{/detalle}.

\imagen{4}{C\'{o}digo de HomeController.java -- el controlador con las rutas del CRUD.}

\subsection{View -- Plantillas HTML con Thymeleaf}

Las vistas son las p\'{a}ginas que uno ve en el navegador. Us\'{e} Thymeleaf para que los datos de la base de datos se muestren en el HTML. Tengo tres: \texttt{index.html} que es la lista, \texttt{form.html} que es el formulario y \texttt{detalle.html} que muestra toda la info de una persona.

\imagen{5}{C\'{o}digo de index.html -- la vista principal con la tabla de individuos.}

\section{Mejoras que Implement\'{e}}

El taller ped\'{i}a tres mejoras espec\'{i}ficas. En la siguiente captura se pueden ver las tres implementadas en la p\'{a}gina principal:

\begin{itemize}[leftmargin=1.5cm, itemsep=3pt]
    \item La columna de \textbf{Tel\'{e}fono} ya aparece en la tabla junto a los dem\'{a}s datos.
    \item Las acciones \textbf{Ver}, \textbf{Editar} y \textbf{Eliminar} est\'{a}n separadas visualmente con delimitadores.
    \item El bot\'{o}n \textbf{``Nuevo Individuo''} est\'{a} visible en la parte superior de la tabla.
\end{itemize}

\imagen{6}{P\'{a}gina principal mostrando las 3 mejoras: columna tel\'{e}fono, acciones separadas y bot\'{o}n Nuevo Individuo.}

Para el tel\'{e}fono me asegur\'{e} de que la entidad tuviera el campo \texttt{numero}, le agregu\'{e} la columna a la tabla del \texttt{index.html} y tambi\'{e}n puse el campo en el formulario. Las acciones las separ\'{e} con barras para que se vean claras y la de eliminar pregunta antes de borrar. El bot\'{o}n de ``Nuevo Individuo'' va a la ruta \texttt{/nuevo}.

\section{Conclusiones}

Con este taller practic\'{e} lo que vimos en clase sobre Spring Boot y la arquitectura MVC. Me qued\'{o} m\'{a}s claro c\'{o}mo se conectan las capas entre s\'{i}: la Entity con la base de datos, el DAO que hace las consultas, el Service con la l\'{o}gica, el Controller que maneja las rutas y las Vistas que muestran todo.

Las mejoras que hice dejaron la app m\'{a}s completa. Ya se ve el tel\'{e}fono, las acciones se distinguen bien y el bot\'{o}n de crear funciona. Lo m\'{a}s importante es que comprob\'{e} que todo lo que hago en la p\'{a}gina web queda guardado en PostgreSQL.
